## Stage 0 — store library, schemas, fixtures — SHIPPED (PR #49)

Build the shared library `scripts/shared/store/` implementing
[the store-core contract](01-store-core.md): zone paths (including the
`state/` zone), atomic write helpers (same-directory temp-and-rename,
line-atomic JSONL append, torn-tail tolerance), the content-addressed blob
store (hash of uncompressed bytes, verify-on-read, reference-count audit),
the manifest envelope (including fetch groups and the over-captured header/
count fields), the build ledger and materialization sequence, index files
with header records, the pinned-key registry, builder-only locking, the
canonical serializer, lowercase-slug enforcement with the neutral-
identifier map, the `store_show.py` entity-to-blob resolver, taxonomy
files, JSON Schemas plus `validate_store.py`, and the (annotations/state-
only) migrations scaffold.

Also: the `data_root` config key; a fully synthetic fixture store under
`examples/data/` (generated by script, `example.com` addresses only); and
unit tests covering crash-injection atomicity, torn-line tolerance, blob
dedup and refcounts, ledger set-difference processing (including the
fetch-started-before-but-committed-after case), idempotent event appends,
**incremental-equals-rebuild equivalence**, byte-identical rebuild
determinism, the annotation-orphan hard-fail, and key-registry pinning.
Plus a leak-guard proof: a seeded personal token under `private/data/`
never reaches the public tree, and the fixture store passes the guard.

Owner decisions folded into this stage (2026-07-21): the fixture-size
check treats 100 KB as a **soft threshold** — exceeding it warns a human
and a human-approved, configurable threshold raise is the sanctioned path
(never a silent grow, never a hard block); slug allocation is
**library-only** with strict write-time pattern validation (agents can
never hand-type an identifier); and every builder/query/validator ships
**missing-raw tolerance** (multi-laptop manual sync: manifest-present,
blob-absent is the normal `not-synced-here` state, tested explicitly).

**Acceptance:** all of the above green in CI; vendoring drift check green;
zero change to any skill's behavior; a fixture over the soft threshold
produces the warning path, not a failure.
**Decisions:** all resolved — see the record above.

## Stage 1 — capture at the jobs fetch boundary — SHIPPED (PR #50)

Wire the capture hook into every job fetch path per
[the capture design](02-job-postings-pipeline.md#1-capture-at-the-fetch-boundary):
the shared-helper sources nearly for free; the four bespoke flows (Workday
multi-request groups, Amazon, Apple's cookie handshake, Meta's GraphQL) as
explicitly costed work items, each writing fetch-group manifests with
honest completeness attestation (only Greenhouse/Ashby/Lever ever attest
complete). Aggregators and JobSpy capture as the scrape tier; JD fetches as
their own operation. Over-capture headers/counts/query-terms from day one —
manifest fields can never be backfilled.

Verify-at-implementation items flagged by research: Greenhouse job
descriptions may arrive HTML-entity-escaped — normalization (itself
versioned) must precede any semantic content hash, or every poll looks
"changed"; confirm SmartRecruiters truncation behavior at its response cap.

Fetchers take no locks; capture failure warns and never fails a search.
The implementation shipped; the remaining soak task is to run real
searches for several days and measure store growth per run
under the honest sizing model, capture overhead (target under one second
per run), dedup ratio on unchanged boards.

**Acceptance:** search output and timing unchanged; raw populates with
valid groups; an induced parse failure demonstrably leaves readable raw
(the design's core promise, as a test); concurrent capture from two
processes is clean.

## Stage 2 — builder, derived postings, index, query tool — SHIPPED (PR #51)

`build_postings.py`: ledger-ordered incremental builds and full rebuilds
(build-aside, verify — schema, counts, 100% annotation matches,
incremental-equals-rebuild spot check — then atomic swap), plus the
opinions-only rebuild that re-runs classifiers over stored facts and prints
the diff. Identity per
[the identity design](02-job-postings-pipeline.md#3-identity): platform-
unique IDs without board tokens, registry migration records, the versioned
URL canonicalizer, weak-identity content keys. **Observations only** —
first-seen / seen / changed events; no closed/disappeared inference in this
stage.

`query_postings.py` with materialization-sequence cursors and compact
output; `store_show.py` wired for jobs; the generated store README
(map + cookbook), registered with the instruction-budget checker; the
**suppressed review queue** (`jobs/index/triage/suppressed-<yyyy-mm>.jsonl`
— one line per suppressed sweep row with partial info + raw manifest path;
write-only from the pipeline, optional manual review, never blocks
anything — owner decision 2026-07-21).

**Acceptance:** determinism and equivalence tests green **against several
days of real Stage-1 raw**, not just fixtures; a deliberate classifier
tweak plus opinions-only rebuild re-labels history with a diff report; the
query examples in the design doc answer with zero network; annotation
orphan count enforced at zero.
**Decisions:** all resolved — suppressed rows get the review queue above;
retention lands as the GC expression config (Stage 4).

**Acceptance:** canaries green; both benchmark rows recorded; snapshot

## Stage 4 — retention, gardener, hardening — SHIPPED (PR #53)

Retention enforcement per
[the retention design](01-store-core.md#9-retention-and-growth), driven by
the owner's **GC expression config** (independent posting-date and
last-observed-date filters, AND default, OR/single supported — see
[the GC config](01-store-core.md#the-gc-config-decided-2026-07-21)):
reference-counted blob deletion, frozen-facts snapshots before pruning
anything a materialized entity depends on, per-manifest tombstones. The
gardener gains the store routine (zone sizes, orphaned blobs, refcount
audit, manifest-less directories, torn tails, stale locks, cursor and queue
ages, annotation-conflict backlog, `not-synced-here` blob counts) and runs
`validate_store.py` over the real store — dry-run by default like all
gardener work. (No at-rest encryption tooling — owner decision: private
machines, user-protected.)

**Acceptance:** gardener dry-run over the real store is clean; a prune
followed by a rebuild degrades exactly as designed (facts carried forward,
zero errors, zero data husks).

## Risks and mitigations


| Risk                                                         | Mitigation                                                                                                                             |
| ------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------- |
| Capture hook destabilizes searches                           | Stage 1 ships alone, lock-free, warn-only; measured for days before anything reads the data                                            |
| Bespoke-fetcher capture (Apple/Meta) costs more than planned | Costed explicitly; can land per-source — the attested cohort (Greenhouse/Ashby/Lever) doesn't wait for them                            |
| Store diverges from reality (builder bug)                    | Determinism + equivalence tests on real raw; annotation-match hard gate; rebuild always available within the stated frozen-facts bound |
| Token savings don't materialize                              | Reframed gates: cold cost-ceiling + warm delta at Stage 3; the big prize measured at Stage 5d, not promised earlier                    |
| Lifecycle fabricates closures of live jobs                   | Closure inference is not built at all (owner decision); the store never claims "closed"                            |
| Email mislink causes a wrong status change                   | Transition-time re-verification is a contract requirement of Stage 5d, tested against the exact attack scenarios                       |
| Scope creep into email before jobs proves out                | Jobs stages 0–4 are complete; Stage 5 is split into focused provider, tracker/calendar, sync, and reconciliation tasks                                                   |
| Disk growth surprises                                        | Stage 1 measures real growth before anything depends on the store; reference-counted retention lands at Stage 4                        |
